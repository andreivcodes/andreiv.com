I haven't manually written code in more than a year. Not a single line.

I created governance platforms that were used to vote on millions of dollars. I did massive refactors of payment flows moving more than $70k every month. I built presentation websites, high-performance blockchain indexers, bots and automations, internal tools like my own accounting software, and even our wedding invitation.

That sounds irresponsible. I understand why it sounds that way. My current view is that it is at least as irresponsible to keep working as if the tooling environment has not changed.

I am not claiming that AI makes software easy, or that it removes judgment, taste, responsibility, or engineering discipline. My claim is narrower: there is a new way to do engineering now, and it is worth trying seriously before dismissing it.

Let's start with a bit of a history lesson.

I want to walk through how I saw this evolve: from the normal chatbot, to playing with system prompts on local models, to thinking models, to thinking with tools, and finally to tools like Claude Code, Codex, and OpenCode, where the model comes to the machine instead of me going to the model.

I do not think this is automatically better in every context. I do think it is real, useful, and easy to underestimate if your picture of AI is still just a chatbot tab.

## Normal Chatbot

The first mainstream product shape was the chatbot.

That matters because it fixed a mental model in place very early. You asked a question. The model replied. You copied the result somewhere else and did the real work yourself.

For a lot of people, that is still what "AI" means. I think that is where part of the gap is right now.

That was already a meaningful shift. Language became a usable interface for software at a scale most people had not experienced before. Summaries, rewrites, brainstorming, code snippets, explanations, all of that became immediate.

But the loop was still narrow. The model stayed inside the chat box. The user carried the output back into the real workflow.

That interface set the baseline mental model: AI is a conversation. Understandable, useful, and already incomplete.

## The System Prompt Was Already There

Even in that early phase, the system was usually not just user input and model output.

By the time chat products became mainstream, many of them already had a higher-priority instruction layer above the visible conversation. OpenAI documents this through system and developer messages, depending on the API and model family. Anthropic exposes the same idea through the system parameter.

This is where the product started to separate from the model in a way most users could not see yet. The same underlying model could behave like a support agent, a tutor, a writing assistant, or a coding assistant depending on the instructions wrapped around the conversation.

I noticed this most clearly when I tried running local models. Once you can touch that layer directly, the illusion breaks quickly. What looked like a single assistant was already a model plus framing, boundaries, tone, and task definition.

That may sound like a small implementation detail, but it was the first real break from the naive chatbot model. The system was no longer just answering your message. It was answering your message inside a set of product-level instructions.

## Thinking Models

The next shift was not mostly about interface. It was about letting the model do more work before it committed to an answer.

In simpler terms, instead of replying immediately, the model could spend some time effectively talking itself through the problem first. That started as a way to improve performance on harder tasks, but it also turned out to be very useful for planning.

OpenAI describes its reasoning models as models trained to think longer and harder about complex tasks. Anthropic approaches the same category through extended thinking, where Claude can produce thinking blocks before the final answer.

This changes the task boundary. Basic chat is often enough for rewriting, retrieval, and first-pass explanation. Thinking models become much more useful when the first answer is likely to be wrong: planning, diagnosis, prioritization, sequencing, and tradeoffs.

  </ConversationMessage>

From the outside, this still looks like chat. Under the hood, it is not just immediate response generation anymore. The system is spending extra inference-time compute to reason before it speaks.

That difference will matter later on.

## Thinking with Tools

Once tool use arrived, the interface still looked like a chatbot, but the operating model changed again.

A tool-using model is no longer limited to what already fits in context. It can ask for a web search, retrieve documents, run code, transform data, or interact with another system. Anthropic's tool-use docs are explicit here: when you provide tools, the API constructs a tool-use system prompt, the model emits tool requests, the surrounding system executes them, returns results, and the loop continues. OpenAI's hosted tools move similar capabilities into the platform through things like web search, file search, and code interpreter.

This is the important distinction: the model does not directly perform the action. It selects the action. The harness around it decides how that action is executed, how the result comes back, and what constraints apply.

That is a bigger shift than it first appears. A plain chatbot answers from prior knowledge and whatever fits in the prompt. A tool-using system can go acquire fresh evidence, transform it, and continue reasoning on top of it.

At that point, calling it just a chatbot starts to miss the mechanics that actually matter.

## Then the Model Came to the Machine

The next shift was not simply better chat products. It was relocation.

Instead of moving work into a browser tab, the model started moving into the environment where the work already lives. In tools like Claude Code, Codex CLI, and OpenCode, the model can see the repository, edit files, run commands, inspect failures, and stay inside the loop until the task is actually resolved.

Programming is unusually well suited to this shift because the medium is already tool-native. The shell, git, linters, test runners, package managers, browsers, logs, CI, and editors are all explicit interfaces. A coding tool does not need a brand new environment to be useful. It can plug into one that already exists.

That does not make the model magically correct. It does make it materially more useful. Instead of asking for advice about the codebase, I can ask it to inspect the codebase. Instead of asking what command I should run, I can let it run the command, see the output, adjust, and continue.

<MachineHarnessFigure caption="The model is no longer answering from outside the workflow. It is operating inside the repository, terminal, and git loop itself." />

This is really the same idea as the system prompt from earlier in the article, but expanded into a working environment. In a normal chatbot, that instruction layer usually sits above the conversation. In a machine-side setup, part of it still starts there, but part of it can also show up during the run itself: tool definitions, permission rules, project instructions, and step-specific reminders injected between actions, constantly guiding the model while it works.

So the model is not running on just my message and the model weights. It is running inside an instruction loop: my request, the current state of the task, the available tools, the permission model, and the rules telling it how to proceed. In practice, that looks less like a single prompt at the top and more like a runtime spec.

<HarnessRuntimeFigure />

This is where chatbot stops being the most useful frame for me. The more accurate frame is a model operating inside a harness environment: tools, permissions, context, feedback loops, project instructions, and interfaces that are all wired into the place where the work already happens.

An agent harness like Claude Code, Codex, or OpenCode is one expression of that. The broader question is how you shape the environment around it so it can actually do a good job on your project: what commands exist, what scripts are available, how the repository is structured, what instructions are persistent, what is safe to automate, and where the boundaries are.

That is what I want to write about next. Not another introduction to chatbots, but a more practical post about the limitations of current harnesses and how I think you can improve them for each of your projects, without vendor lock-in and by leaning on the tools you already have.
